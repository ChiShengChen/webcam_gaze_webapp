# Webcam-gaze benchmark — 2026-06-08 session

Paper §4.4 cross-engine data: accuracy + latency on commodity hardware (2020-era MacBook). Two engines compared:

- **WebGazer** — baseline. Ridge regression on raw eye-patch pixels.
- **FaceMesh+KRR** — this work. Kernel ridge regression on 13-dim feature derived from MediaPipe FaceMesh landmarks.

All runs share the same physical setup: single light source, fixed seating, fixed viewing distance ~60 cm, DevTools closed, no window resize, ≥30 s rest between runs.

---

## Wiring fixes applied before data collection

Two latency-measurement bugs fixed first so the inference/pipeline-latency columns are meaningful.

### Fix 1 — FaceMesh capture clock (commits cf70502, addc947)

Pre-fix: `captureTime` was set to emit time, so `inference_latency = emit − capture ≈ 0`. Post-fix: rVFC `presentationTime` flows through a per-frame queue in [src/gaze/landmarks.ts](../src/gaze/landmarks.ts), surfaced to the controller via [src/gaze/engine.ts](../src/gaze/engine.ts).

### Fix 2 — WebGazer capture clock (this session)

WebGazer's library API doesn't expose per-frame timestamps. We run a parallel rVFC loop on `webgazerVideoFeed` and tag each gaze sample with the most-recent `presentationTime`. See [src/main.ts](../src/main.ts) (`startWebgazerCaptureClock`).

**Approximation caveat:** WebGazer's internal frame queue depth is opaque, so the real source frame may be 1–2 frames older than the value we record. The reported inference latency is therefore a **lower bound** on the engine's true inference time. On browsers without rVFC (older Safari) both engines fall back to `captureTime = emitTime` and the column reads 0.

## 4-run paper matrix — pursuit calibration

**Protocol differs by task — sweep ≠ drift, cross-task comparisons are not apples-to-apples.**

| Task | Grid | Dwell | Visit strategy | URL pattern |
|---|---|---|---|---|
| sweep | 16 × 8 (128 cells) | 3 s | Row-major, all cells | `?task=sweep` |
| drift | 12 × 8 (96 cells) | 2 s | Random subset of `visits` cells, idle gap between visits (drift signal lives in the gap) | `?task=drift&visits=N` |

Within each task, engine comparison is valid (#1 vs #2 for sweep; #3 vs #4 for drift). Across tasks, only same-engine longitudinal comparisons are interpretable.

## Per-cell spatial structure (heatmap findings)

The four heatmaps (`heatmap_*.png`) reveal that **both engines have spatial structure in their error — but the patterns are qualitatively different**. This was not predicted by the aggregate mean / median numbers; it only shows up at per-cell granularity.

### FaceMesh — radial gradient (center good, all corners bad)

In the sweep heatmap, FaceMesh's error grows roughly **radially** from a central low-error region:

- Best cell: row 0 col 10, **1.9 °** — near the paper's ≤ 2 ° target.
- A clear central band (rows 0–4, cols 5–13) sits at 3–8 °.
- All four corners exceed 18 °; the worst single cell (row 4 col 0) is 32 °.
- Bottom band (rows 5–7) is systematically worse than top — likely the camera's vertical FOV cutting off iris landmarks at downward gaze, but worth checking against `tracking_loss_pct` in a future ablation.

This is the canonical **calibration extrapolation failure**: KRR with finite training samples can only interpolate well inside the convex hull of the calibration targets, and the pursuit Lissajous likely under-covered the screen corners.

### WebGazer — diagonal gradient (bottom-left good, top-right bad)

In the sweep heatmap, WebGazer's low-error region is in the **bottom-left**, with error rising monotonically toward the top-right:

- Best cell: row 5 col 1, **2.8 °** (comparable to FaceMesh's best!).
- Bottom-left quadrant (rows 4–7, cols 0–4) sits at 3–10 °.
- Top-right quadrant (rows 0–2, cols 8–15) is uniformly 18–30 °.
- Worst single cell (row 7 col 15): 35.9 °.

This is _not_ a radial extrapolation pattern — it's a **directional bias**, the kind of failure mode that comes from one of:
- Asymmetric coverage during pursuit calibration (Lissajous spent more time in the bottom-left of the screen than the top-right),
- Head-pose asymmetry that the ridge model on raw eye patches can't disentangle from gaze direction,
- Or a quirk of WebGazer's eye-patch features that's more sensitive to one quadrant.

The diagonal pattern is reproducible across the WebGazer pursuit baselines in [gaze_result/](.) — worth investigating before paper submission whether it generalises across users / sessions or is specific to this setup.

### Why the original "WebGazer = uniform noise" reading was wrong

The aggregate numbers misled us. WebGazer's 3×3 region breakdown (center 12.30 ° / edge 15.61 ° / corner 16.97 °) has a *shallower* gradient than FaceMesh's (6.37 ° / 10.40 ° / 13.83 °), which we initially read as "less structured." But the 3×3 partition averages over all four corners, smearing out WebGazer's diagonal bias — the per-cell view shows the structure is real, just not radially symmetric. **Take-away:** the 3×3 region table is sufficient for FaceMesh but lossy for WebGazer; the full heatmap is needed to see the WebGazer pattern.

## Notes for paper write-up

- **Apples-to-apples within task:** both engines tested at identical protocol per task (sweep = 16×8 / 3 s; drift = 12×8 / 2 s / 10 visits) in the same session with consistent posture and lighting. Engine comparisons within a task are valid; cross-task comparisons are not.
- **FaceMesh full-density vs fast=1:** earlier FaceMesh fast=1 runs (8×4 grid) showed 8.8 °. Full 16×8 adds corner cells that calibration cannot extrapolate to well, raising the mean to 11.78 °. The full-density number is paper-relevant; do not quote the fast=1 figure as the headline accuracy.
- **Latency story:** FaceMesh p95 27 ms passes the sub-50 ms target from [gaze_v2/PLAN.md](../gaze_v2/PLAN.md); WebGazer p95 52 ms is marginal (the rVFC approximation is a _lower bound_ — true WebGazer p95 may be ≥ 52 ms by 1–2 frame intervals).
- **Accuracy-precision tradeoff is a real finding:** FaceMesh wins on mean error (28–46 % lower) but loses on within-fixation jitter velocity (p99 1.6–3.5× higher than WebGazer). The One-Euro filter doesn't fully smooth FaceMesh's per-frame iris-landmark micro-movements. This is worth a paragraph in §4.4 — it complicates the simple "FaceMesh dominates" narrative.
- **No observable calibration drift:** both engines show _negative_ drift rates over 4.5 min drift runs (error decreases over time). Reading: posture stabilization swamps any model decay. Paper can claim "no measurable accuracy drift over 4–5 min sessions on a 2020-era MacBook," which is a useful baseline result.
- **Spatial-structure asymmetry needs the heatmap, not the 3×3 partition:** the 3×3 region table (center / edge / corner) understates WebGazer's structure because it averages across the four corners, smearing out the diagonal bias. The full per-cell heatmap is the right figure for §4.4 — show all four, side-by-side, with a shared colour scale (we used vmax = 20 ° for cross-comparability).

| retention % | 52 | 67 | 41 | 63 |

There is no monotonic, U-shaped, or otherwise interpretable signal as a function of β. Instead, accuracy and samples retention are tightly correlated _across_ runs ($r \approx 0.97$): runs with high retention are accurate, runs with low retention are not. Calibration-time γ also varies a lot across these four runs (8.88e-2 → … → 1.31e-1), which is set by a median-pairwise-distance heuristic on the calibration feature distribution — γ varying means the feature distribution itself moved between calibrations, i.e., the user's head pose / lighting drifted between runs more than the β knob meaningfully altered.

**Honest conclusion for the paper:** at $N=1$ per condition, between-run posture / lighting variance ($\sim 7^\circ$ on this protocol) dominates whatever effect β has in the tested range $\{0.003, 0.007, 0.015, 0.030\}$. The β ablation cannot draw a parameter-recommendation curve from this data and should be reported as a methodological observation (the harness exposes the noise floor) rather than a finding about One-Euro tuning. A multi-replicate (≥3 per condition) follow-up is the proper fix.

This recasts the §5.4 accuracy / precision tradeoff finding too: the within-fixation $v_{p99}$ split between WebGazer and FaceMesh in Table 1 may itself be partially attributable to engine-coupled posture noise. Worth flagging as a limitation rather than as a tradeoff prescription.

#### Run A4 — kernel = linear

- CSV: [benchmark_facemesh_pursuit_abl-oneB0-k-linear_2026-06-08-13-47-32.csv](benchmark_facemesh_pursuit_abl-oneB0-k-linear_2026-06-08-13-47-32.csv)
- Gazemap PNG: [gazemap_facemesh_pursuit_abl-oneB0-k-linear_2026-06-08-13-47-32.png](gazemap_facemesh_pursuit_abl-oneB0-k-linear_2026-06-08-13-47-32.png)
- URL: `/?engine=facemesh&task=sweep&kernel=linear`
- Samples: 7 908 (69 % retention) across 128/128 cells, sample rate 30.1 Hz, tracking loss 0 %
- **Mean error:** 16.83 ° (vs baseline 11.78 °, **+43 %**)
- **Median error:** 15.28 ° (vs 11.54 °)
- **Hit rate (1.5 °):** **7.81 %** (vs 2.34 % — **3.3× higher** despite worse mean)
- **Inference / pipeline latency:** 22 / 22 ms median; 27 / 28 ms p95 (identical to RBF baseline — kernel choice doesn't affect inference cost at this calibration size)
- **KRR fit diagnostics:** N=463 calibration samples, γ=1.05e-1, λ=1.0e-3
- **Naming note:** filename includes a spurious `oneB0-` tag because the `Number(null) === 0` fallthrough in [src/main.ts](../src/main.ts)'s URL parser triggered the ablation suffix even though `onebeta` wasn't passed. Data inside the CSV is correct (`kernel=linear` in diagnostics). Fix the parser after the sweep finishes to avoid filename pollution.

**Reading — linear kernel produces bimodal error distribution.** The mean-up / hit-rate-up combination ($16.83^\circ$ mean alongside $7.81$ % hit rate) is mechanistically diagnostic. RBF localises outside the calibration convex hull (predictions → target mean), producing a relatively uniform but moderately high error everywhere. Linear KRR collapses to ridge regression on the 13-dim feature space and extrapolates linearly outside the hull — central cells (in-distribution) get hit cleanly under $1.5^\circ$ while corner cells (out-of-distribution) fly out of bounds, dragging the mean.

This is the first ablation finding in this session that is **structurally interpretable** rather than noise-dominated. Crucially, the mechanistic prediction is testable: a per-cell heatmap for the linear-kernel run should show a much sharper centre-versus-corner contrast than the RBF heatmap of Fig.~3. Worth generating with `bench/heatmap.py --vmax 20` on this CSV after the sweep finishes.

The hit-rate finding has a useful corollary for downstream consumers of the gaze signal: if the application only consumes gaze in the central viewport region (e.g., region-prompting SAM where the surgeon looks primarily at the central operative field), linear KRR could be competitive with RBF at a much lower training-time cost (linear KRR with explicit feature mapping is just ridge regression, $O(d^3)$ instead of $O(N^3)$). Out of scope to evaluate here; flagged as future work.

#### Run A5 — kernel = poly2

- CSV: [benchmark_facemesh_pursuit_abl-oneB0-k-poly2_2026-06-08-13-57-57.csv](benchmark_facemesh_pursuit_abl-oneB0-k-poly2_2026-06-08-13-57-57.csv)
- Gazemap PNG: [gazemap_facemesh_pursuit_abl-oneB0-k-poly2_2026-06-08-13-57-57.png](gazemap_facemesh_pursuit_abl-oneB0-k-poly2_2026-06-08-13-57-57.png)
- URL: `/?engine=facemesh&task=sweep&kernel=poly2`
- Samples: 4 956 (43 % retention) across 128/128 cells, sample rate 30.1 Hz, tracking loss 0 %
- **Mean error:** 28.08 ° (vs RBF baseline 11.78 °, **+138 %**; vs linear A4 16.83 °, **+67 %**)
- **Median error:** 22.15 °
- **Hit rate (1.5 °):** 1.56 % (vs RBF 2.34 %, linear 7.81 % — **lowest of the three**)
- **Inference / pipeline latency:** 22 / 22 ms median; 27 / 28 ms p95 (identical to others)
- **KRR fit diagnostics:** N=465 calibration samples, γ=1.01e-1 (reported but unused — poly2 ignores γ), λ=1.0e-3

**Reading — poly2 falsifies the "locality continuum" hypothesis from A4.** If the dominant axis were kernel locality, poly2 (a smooth degree-2 polynomial with weak locality) should land between linear and RBF. Instead poly2 is substantially \emph{worse} than both, with a distinct combination of: high mean, low hit rate, and low samples retention. The retention drop (43 %, half of linear's 69 %) implies the I-VT classifier sees many more saccades on the poly2 pipeline's gaze output — i.e., poly2's per-frame predictions are intrinsically noisier than the other two.

Mechanistic explanation: a degree-2 polynomial kernel $K(a,b) = (a \cdot b + 1)^2$ on 13-dim standardised features induces an explicit feature expansion to ${\sim}91$ dimensions (all monomials of degree $\le 2$). At our calibration size $N \approx 465$, that is ${\sim}5$ samples per expanded dimension — marginal for the regularisation strength $\lambda = 10^{-3}$ tuned on the much-smaller-effective-dimension RBF. The diagonal-vs-off-diagonal scale of the poly2 kernel matrix is also far more extreme than RBF's: $K(x,x) = (\|x\|^2 + 1)^2 \approx 196$ for a 13-dim standardised vector, while off-diagonal entries are $O(1)$. Cholesky succeeds but the implied effective regularisation is much weaker than the same nominal $\lambda$ buys on RBF. Either of these would produce the observed high-variance prediction.

Three corrected mechanistic stories, one per kernel:

| Kernel | Out-of-distribution behaviour | Predictions | Samples retention | Accuracy structure |
|---|---|---|---|---|
| RBF | localises (→ training centroid via target centring) | moderate, smooth | 67 % | uniform moderate error |
| linear | linear extrapolation | clean in-dist, far-flying OOD | 69 % | bimodal (centre good, corner catastrophic) |
| poly2 | unregularised feature expansion at small $N$ | high variance everywhere | 43 % | uniformly noisy |

The defensible claim is therefore not "non-linear $>$ linear" (poly2 falsifies that) but the sharper:

> The RBF kernel's locality property — predictions revert to the training centroid out-of-distribution — produces better full-screen behaviour than either linear ridge or polynomial expansion at the regularisation strength tuned for RBF. This locality is what produces the radial gradient observed in Fig.~3 §5.3.

The mechanistic prediction made in A4's reading (linear should produce a sharper centre-vs-corner heatmap than RBF) still stands and is testable on the released CSVs; an analogous prediction holds for poly2 (its heatmap should look noisy without a clear centre-versus-corner structure). Both can be verified with `bench/heatmap.py --vmax 30` post-hoc.

**Confounders that remain unresolved at $N=1$:** poly2's high error is partly attributable to using RBF-tuned $\lambda$ on a kernel with very different conditioning. A clean kernel comparison would tune $\lambda$ separately per kernel (e.g., LOOCV per kernel) — we did not. Future work.
